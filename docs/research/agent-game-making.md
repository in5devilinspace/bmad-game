# Deep Research: AI-Agent-Driven Mini Game Creation

*106-agent research fleet, adversarially verified (3-vote refutation per claim). 2026-07-06.*

## Question

Agent skills and techniques for AI-agent-driven mini game creation: (1) existing Claude Code / agent skills, plugins, and slash commands for making, rendering, and generating mini games; (2) rendering approaches for agent-built games that are more efficient and richer than terminal output (HTML5 canvas, WebGL, SVG, artifacts); (3) patterns where AI agents ARE characters inside a game (LLM-driven NPCs, agent-controlled entities, generative agents like the Stanford Smallville paper); (4) best practices for multi-agent pipelines (BMAD-style) that autonomously design and ship playable games.

## Summary

A mature, verified ecosystem now exists for AI-agent-driven mini-game creation across all four axes of the question. (1) Concrete Claude Code tooling ranges from single subagent definitions (awesome-claude-code-toolkit's game-developer) to full studio simulations (Claude-Code-Game-Studios: 49 agents + 73 skills in a three-tier Opus/Sonnet/Haiku hierarchy) and domain-specific skill packs (threejs-game-skills' director-plus-specialists pipeline). (2) Rendering has converged on single-file, zero-build, self-contained HTML opened in a browser — an officially documented Claude Code skills pattern — realized via canvas libraries (crisp-game-lib), Three.js/WebGL, and Phaser, with agent-side verification via Playwright screenshots and canvas nonblank-pixel checks. (3) LLM agents as in-game characters is proven at three maturity levels: research believability (Stanford Smallville's memory/reflection/planning architecture), open-ended embodied competence (Voyager's curriculum + skill library + iterative prompting in Minecraft), and live commercial deployment (KRAFTON's PUBG Ally co-playable character, running a behavior-tree System 1 plus on-device 2B SLM System 2). (4) Multi-agent pipelines span a deliberate autonomy spectrum — from human-approval-gated studio hierarchies to fully zero-touch loops where a single 'Create a game' prompt triggers design, implementation, genetic-algorithm-based automated playtesting, and up to three improvement iterations — with recurring best practices of tiered role/model assignment, director-based skill routing, evidence-before-success verification bars, and playable-loop-first quality gates.

## Verified Findings

### 1. (high confidence)

**Claim.** Claude-Code-Game-Studios is the largest existing Claude Code game-development toolkit: 49 specialized agents and 73 workflow skills (the slash-command surface) organized as a three-tier studio hierarchy with per-tier model assignment (Tier 1 Directors on Opus: creative-director, technical-director, producer; Tier 2 Department Leads on Sonnet: game-designer, lead-programmer, art-director, etc.; Tier 3 Specialists on Sonnet/Haiku) — and it explicitly rejects full autonomy via a five-step protocol (Ask → Present options → User decides → Draft → Approve) so no artifact is written without human sign-off.

**Vote:** 3-0 on all three constituent claims [0][1][2]

**Evidence.** Verified beyond the README via GitHub trees API: exactly 49 agent .md files in .claude/agents/ and 73 SKILL.md files in .claude/skills/; model assignments confirmed in agent frontmatter (creative-director/technical-director/producer declare model: opus; game-designer/lead-programmer/art-director/qa-lead declare model: sonnet; Tier 3 mixed sonnet/haiku). The approval protocol is operationalized in CLAUDE.md and per-agent files (e.g., engine-programmer.md: 'You are a collaborative implementer, not an autonomous code generator'), with no auto-approve bypass in the 543-file tree. 22,666 stars, created 2026-02-12, pushed 2026-05-21.

Sources: https://github.com/Donchitos/Claude-Code-Game-Studios

### 2. (high confidence)

**Claim.** Fully autonomous end-to-end mini-game creation with automated playtesting is demonstrated by abagames/claude-one-button-game-creation: Claude agents design rules, implement browser-playable games in crisp-game-lib (~150 lines each, single tap/hold/release input), evaluate them with a genetic-algorithm simulation harness (ga_tester.js, GA ratio = GA best score vs monotonous-input score, pass threshold >1.5), and iterate on weak designs up to 3 times — the human only issues 'Create a game.'

**Vote:** 3-0 on all three constituent claims [5][6][7]

**Evidence.** Verified at repo HEAD (SHA 803bcdc, pushed 2026-06-28): ga_tester.js exists with GA_CONFIG {populationSize: 50, generations: 30}; for_zero_touch/CLAUDE.md defines Phase 4 GA evaluation and Phase 5 improvement ('iterates up to 3 times'); 16 example games 'generated by Claude Code (Claude Opus 4.5) in response to a single Create a game request, with no human intervention.' Spot-checked game files measured 103-251 lines, clustering near 150. Author (Kenta Cho) is crisp-game-lib's own author.

Sources: https://github.com/abagames/claude-one-button-game-creation · https://github.com/abagames/claude-one-button-game-creation/blob/main/archive/for_zero_touch/README.md · https://github.com/abagames/claude-one-button-game-creation/blob/main/CLAUDE.md

### 3. (high confidence)

**Claim.** threejs-game-skills implements a director-plus-specialists skill pipeline for WebGL games: a single 'threejs-game-director' skill routes broad requests to specialists (gameplay, AAA graphics, UI, QA/release, 3D/image/audio asset generation) so users never name skills manually; it enforces an evidence-before-success verification bar (npm build, browser preview, console error check, Playwright screenshot, canvas nonblank-pixel check, desktop + mobile viewports) and a quality bar requiring a playable game loop first ('a static scene is not done') with premium/AAA claims gated by a 0-3 visual scorecard where any category below 2 disqualifies the build.

**Vote:** 3-0 on all three constituent claims [8][9][10]

**Evidence.** All quotes verified verbatim in AGENTS.md (SHA ee3c47c); director's SKILL.md contains explicit Phase Routing and Mandatory Sibling Skill Loading sections; the canvas pixel check is operationalized in skills/threejs-qa-release/scripts/inspect-threejs-canvas.mjs; the scorecard is a real 10-category rubric at skills/threejs-aaa-graphics-builder/references/visual-scorecard.md (every category >=2, average >=2.3, automatic-failure list). Repo: 766 stars, created 2026-06-14, pushed 2026-06-16. Nuance: on Claude-style runners the director reads sibling SKILL.md files into context rather than truly invoking them; 'enforce' means prompt-level instruction, not CI gating.

Sources: https://github.com/majidmanzarpour/threejs-game-skills/blob/main/AGENTS.md · https://github.com/majidmanzarpour/threejs-game-skills

### 5. (high confidence)

**Claim.** The dominant rendering approach for agent-built games richer than terminal output is the single-file, zero-build, self-contained HTML document: Anthropic's official Claude Code skills docs document the pattern of skills bundling scripts that generate interactive HTML files opening in the browser, and the dogum/html-artifacts skill (Apache 2.0, v0.1.0 May 2026, ~121 stars) enforces exactly this — one .html file, inline CSS in <style> and JS in <script>, no bundler or npm install — which matches the CSP-sandboxed constraints of artifact viewers where agent-built mini games are typically displayed.

**Vote:** 3-0 on all three constituent claims [3][11][12]

**Evidence.** Official docs verified live (2026-07-06): 'One powerful pattern is generating visual output: interactive HTML files that open in your browser,' with a worked codebase-visualizer example generating 'a self-contained HTML file.' html-artifacts SKILL.md rule 1 verified verbatim: 'Single self-contained .html file. No build step, no bundler, no npm install,' plus explicit Claude.ai artifact-environment rules (no localStorage, single file, restricted external scripts). The mini-game applicability is a mild, clearly-framed inference — an HTML5 canvas game is precisely this kind of interactive HTML output, as the crisp-game-lib and Three.js findings demonstrate in practice.

Sources: https://code.claude.com/docs/en/skills · https://github.com/dogum/html-artifacts

### 6. (high confidence)

**Claim.** The Stanford generative agents paper (Park et al., UIST 2023) is the canonical pattern for LLM agents as game characters: 25 LLM-driven agents populate the Smallville sandbox town (inspired by The Sims, rendered with the Phaser HTML5 web framework), with believability produced by a three-part architecture — natural-language memory stream, higher-level reflection, and retrieval-based planning — that ablations show each contribute critically; agents interface with the game via a server maintaining a JSON structure of each agent's location, current action, and interacting object, and from a seeded intention to throw a Valentine's Day party, agents autonomously spread invitations, formed acquaintances, asked each other on dates, and coordinated attendance over two simulated days.

**Vote:** 3-0 on [13][14][16]; 2-1 on [15] (Valentine's party seeding nuance)

**Evidence.** All abstract quotes verified verbatim against the peer-reviewed primary source (UIST 2023 Best Paper); Phaser + JSON server architecture confirmed both in the paper's full text and in the released implementation (Phaser 3.55.2 in environment/frontend_server templates, Django environment server paired with reverie.py agent server). Party diffusion measured: awareness spread 4%→52% of agents over two game days; five agents coordinated arrival. Caveat on the party experiment (the one 2-1 vote): the paper's own discussion notes TWO user-seeded facts (Isabella's party intent AND Maria's crush on Klaus), so 'single seed' is the authors' slightly generous abstract framing, and only 5 of 12 invitees attended.

Sources: https://dl.acm.org/doi/fullHtml/10.1145/3586183.3606763 · https://arxiv.org/abs/2304.03442 · https://github.com/joonspk-research/generative_agents

### 7. (high confidence)

**Claim.** Voyager (Wang et al., 2023, TMLR) demonstrates an LLM agent as an autonomous embodied in-game entity: presented as the first LLM-powered embodied lifelong learning agent in Minecraft, it explores, acquires skills, and makes discoveries without human intervention using three reusable components — an automatic curriculum maximizing exploration, an ever-growing skill library of executable code, and an iterative prompting mechanism incorporating environment feedback, execution errors, and self-verification — all driven purely by black-box GPT-4 queries with zero fine-tuning or parameter updates.

**Vote:** 3-0 on all three constituent claims [17][18][19]

**Evidence.** All three architecture components and the 'first LLM-powered embodied lifelong learning agent' framing verified verbatim in the abstract; no credible source disputes the firstness framing (prior LLM Minecraft agents DEPS/Plan4MC lacked the lifelong skill library). Qualifications found but not contradicting: GPT-4 does not do pixel-level control — generated code runs on Mineflayer's hand-coded high-level primitives (pathfinder, mineBlock), so off-the-shelf LLM APIs suffice only given a high-level game-side action API; GPT-3.5 performs far worse; vanilla Voyager lacks skill refinement/pruning (addressed by follow-on work like SAGE/EvoSkill). Follow-on literature confirms the skill-library pattern's reusability.

Sources: https://arxiv.org/abs/2305.16291

### 8. (high confidence)

**Claim.** The current academic consensus framework for LLM-based game agents (LLMGAs) is the unified reference architecture of the survey arXiv 2404.02039 (v5, June 2026): single-agent designs organize around exactly three core components — memory, reasoning, and perception-action interfaces — that jointly characterize how an LLM perceives, thinks, and acts in a game, while multi-agent systems achieve coordination, role differentiation, and large-scale social behaviors through two mechanisms: communication protocols and organizational models (covering generative-agent societies like Smallville, Werewolf, and Diplomacy).

**Vote:** 3-0 on both constituent claims [20][21]

**Evidence.** Both abstract sentences verified verbatim against the live v5 abstract (revised 2026-06-08, one month before verification); the survey's PDF cites park2023generative, Werewolf, Diplomacy, and Sotopia, confirming coverage of generative-agent societies. Caveat: the original v1 (April 2024) abstract described SIX components (perception, memory, thinking, role-playing, action, learning); the three-component framing is the current revision, so citations pinned to older versions will differ. Actively cited by follow-up work; no disputes found.

Sources: https://arxiv.org/abs/2404.02039

### 9. (high confidence)

**Claim.** LLM-driven in-game characters have reached live commercial deployment: KRAFTON's PUBG Ally — framed as a 'co-playable character' (CPC), a category distinct from NPCs — ran a public beta in PUBG: BATTLEGROUNDS Arcade Mode June 17 to ~June 30/July 1, 2026, using a dual-layer architecture that is the concrete pattern for real-time LLM game agents: a behavior tree ('System 1') handles movement, aiming, and combat at game tick rate while a language model ('System 2') handles intent interpretation, coordination, and speech — with the entire pipeline (NVIDIA ACE ASR, quantized Mistral-NeMo-Minitron-2B SLM, custom in-house TTS) running on-device rather than in the cloud.

**Vote:** 3-0 on all three constituent claims [22][23][24]

**Evidence.** All quotes verified verbatim in the NVIDIA developer blog (2026-06-25); independently corroborated by KRAFTON's own press release and engineering blog ('From Workflow-Based SLM to Autonomous Agent'), a GDC 2026 session, GamesBeat, and PC Guide (local RTX GPUs with >=8GB VRAM, confirming on-device inference). Discrepancies/nuances: KRAFTON says beta ends July 1 vs NVIDIA's June 30 (one-day, likely timezone); 'LLM-driven' is technically a 2B-parameter small language model; gameplay reviews criticize execution quality (mis-marking enemies) without contradicting the architecture; the NVIDIA source is marketing-adjacent but the architectural facts are first-party corroborated.

Sources: https://developer.nvidia.com/blog/how-krafton-built-pubg-ally-a-co-playable-character-powered-by-nvidia-ace/ · https://krafton.ai (KRAFTON engineering blog, 2026-04-15)

### 10. (high confidence)

**Claim.** Multi-agent game-creation pipelines occupy a deliberate design spectrum on autonomy, with shared best practices at both ends: human-gated (Claude-Code-Game-Studios: mandatory approval before any write) versus fully autonomous (abagames zero-touch: one prompt to shipped game), and threejs-game-skills in between (autonomous execution, machine-verified evidence). The recurring best practices across all three are: (a) tiered role hierarchy with model-cost matching (expensive models for direction, cheap for specialists), (b) a single director/orchestrator skill that routes so users never micromanage specialists, (c) evidence-before-success verification (builds, screenshots, pixel checks, simulated play) replacing agent self-report, (d) playable-loop-first quality gates with numeric scorecards, and (e) automated playtesting as the substitute for the human in fully autonomous loops.

**Vote:** Derived from 3-0 claims [0][1][2][5][7][8][9][10][21]

**Evidence.** Synthesis across three independently verified primary sources, each 3-0 confirmed. The autonomy-spectrum framing is interpretive but each endpoint is the project's own explicit stated design ('This is not an auto-pilot system' vs. 'no human intervention'). The LLMGA survey's communication-protocols + organizational-models framing generalizes practice (a)/(b). Note that in all three toolkits, 'enforcement' of gates and verification bars is prompt-level instruction rather than mechanical CI blocking.

Sources: https://github.com/Donchitos/Claude-Code-Game-Studios · https://github.com/abagames/claude-one-button-game-creation · https://github.com/majidmanzarpour/threejs-game-skills/blob/main/AGENTS.md · https://arxiv.org/abs/2404.02039
